Web 安全不是某一个库的单独开关。四个仓库各守一层边界，应用仍需负责身份、授权、业务校验和部署策略。

## 分层责任

| 层 | 已提供 | 应用仍需负责 |
| --- | --- | --- |
| `yanxu-html` | 文字/属性转义、名称白名单、危险 URL 协议拒绝、显式原始内容。 | CSP、可信内容来源、CSS/脚本/SVG策略。 |
| `yanxu-http` | 严格请求行/首部、长度预算、首部注入防护、安全 Cookie 约束。 | TLS、随机会话、签名/加密、速率限制、可信代理。 |
| `yanxu-web` | 404/405、统一错误、返回类型验证、静态路径防穿越。 | 认证授权、日志脱敏、CSRF策略、输入领域校验。 |
| `yanxu-webblog` | 安全首部示例、转义回归、跨库集成测试。 | 持久存储、账户系统、备份、生产运维。 |

## 当前服务器模型

0.1 只实现 UTF-8 首部的 HTTP/1.1，每个连接读取一个请求，响应固定`Connection: close`。服务器串行接受连接，不支持请求流、并发工作池、背压、优雅关闭或长连接复用。

它适合本地开发、教学、协议基线和集成验证。即使放在反向代理后，后端串行容量和慢请求风险仍然存在，因此不应把当前版本描述为通用生产服务器。

## 使用原始 HTML

`HTML.原始`是显式信任升级，不是清洗器。代码审查应搜索所有原始节点，确认输入是源码常量、可信结构化生成结果或经过上下文正确的专用清洗器处理。用户、接口、数据库和文件数据默认全部进入`HTML.文字`或文字属性。

## Cookie 与会话

Cookie 默认`HttpOnly`和`SameSite=Lax`，但回环开发所需的`Secure`默认关闭。HTTPS 部署必须显式打开`Secure`。本库不生成会话标识；应用应使用安全随机源，必要时进行签名或服务端存储，并建立轮换、过期和撤销机制。

## 演进阶段

后续能力计划按可独立验收的阶段进入：

1. 请求分块传输：明确长度优先级、分块扩展/尾部处理与总预算；
2. Multipart 文件上传：文件数、单文件/总量、路径与临时文件生命周期；
3. 持久连接复用：请求边界、空闲超时、最大请求数与优雅关闭；
4. WebSocket：握手、帧预算、掩码、关闭状态与背压；
5. HTTP/2：帧层、流并发、HPACK、流控与降级互操作；
6. HTTP/3：QUIC、证书、0-RTT策略、QPACK与迁移行为。

顺序可能随言序异步运行时和包生态进展调整，但每项能力都需要规格测试、模糊/边界测试和真实客户端互操作后才进入稳定接口。

## 生产化还缺什么

协议支持之外，生产服务还需要并发隔离、取消与超时传播、结构化日志、指标与追踪、健康/就绪探针、优雅关闭、配置/密钥管理、TLS、可信代理策略、速率限制和安全发布流程。它们会在基础协议稳定后分层加入，不会塞入一个难以测试的单体服务器。

当前可运行基线是[yanxu-webblog](/web/webblog/)；四个仓库的实现进度和问题应分别在对应 GitHub 仓库跟踪。
